# Miku 0.0.3 workflow simulation

This document pretends the ADR-0017 system already exists. It tests the data
model and user workflows before implementation so missing boundaries become
visible early.

## Simulated vault

```text
vault/
  00-inbox/
    capture-2026-07-15.md
  projects/miku/
    home.md
    backend.md
    frontend.md
  concepts/
    files-as-truth.md
    note-workspace.md
  references/
    trilium.md
```

### Notes and placements

| Note | Source file | Placements |
| --- | --- | --- |
| Miku Home | `projects/miku/home.md` | Miku / Projects |
| Rust Backend | `projects/miku/backend.md` | Miku / Projects, Concepts / Architecture |
| Note Workspace | `concepts/note-workspace.md` | Concepts |
| Files as Truth | `concepts/files-as-truth.md` | Concepts, Miku / Principles |
| Trilium Reference | `references/trilium.md` | References |

The same Markdown file appearing in two tree locations is a placement clone,
not duplicated content.

## Simulation 1 — startup and tree selection

### Input

The user opens Miku at `/workspace`.

### Expected behavior

1. Rust returns workspace bootstrap data and the first tree level.
2. The frontend opens the last workspace note if it still exists.
3. The selected note is highlighted in every visible placement.
4. Its ancestor branches are expanded.
5. The center pane loads the note.
6. The right panel loads backlinks, outgoing links, attributes, and children.

### Result

Works if selection is keyed by `note_id`, not by tree row ID. A cloned note must
highlight both placements while retaining one active placement for “reveal in
tree”.

### Gap discovered

The UI needs two identifiers:

```text
note_id       content identity
placement_id  selected tree occurrence
```

Using only `path` or only `note_id` cannot correctly implement cloned tree
selection.

## Simulation 2 — opening a cloned note

### Action

The user selects `Rust Backend` under `Concepts / Architecture`.

### Expected behavior

- the note opens once in the center workspace;
- the active placement becomes `Concepts / Architecture`;
- breadcrumbs show the active placement;
- the other placement remains available under `Projects / Miku`;
- edits modify only `projects/miku/backend.md`;
- both tree rows reflect the same updated title/status.

### Result

Works with a separate `Placement` model.

### Gap discovered

Context actions must declare whether they operate on:

- the note (`edit`, `bookmark`, `archive`, `revision`), or
- the placement (`remove from tree`, `reorder`, `move`, `clone`).

This distinction must be explicit in the API and UI confirmation text.

## Simulation 3 — editing and indexing

### Action

The user changes `backend.md` and adds:

```markdown
[[Note Workspace]]
#rust
```

### Expected workflow

```text
editor dirty
  → save request
  → Rust validates note identity and revision hash
  → atomic file write
  → save acknowledged
  → watcher emits event
  → indexer reparses backend.md
  → backlink appears on Note Workspace
  → rust tag index contains rust
  → frontend receives workspace update
```

### Result

Works, but the UI needs separate states:

```text
dirty → saving → saved-to-file → indexing → indexed
```

“Saved” cannot mean “all derived context is already updated.”

## Simulation 4 — external edit

### Action

The user edits `backend.md` in Neovim while Miku is open.

### Expected behavior

1. The watcher identifies the changed file.
2. If the note is not dirty in Miku, the editor refreshes safely.
3. If Miku has unsaved changes, the frontend shows a conflict state.
4. The user can reload external content, keep local changes, or open a diff.

### Gap discovered

The write contract needs a revision token containing at least:

```text
source_path + content_hash + mtime
```

Path alone is not enough for conflict detection.

## Simulation 5 — rename and stable identity

### Action

The user renames `backend.md` to `rust-backend.md` through Miku.

### Expected behavior

- the Markdown file moves atomically;
- its `id` remains unchanged;
- placements remain attached to the note;
- incoming wikilinks continue resolving by stable identity or updated alias;
- old path links are reported as redirects or broken links;
- the index removes the old path and adds the new path;
- open tabs follow the note to the new path.

### Gap discovered

Stable IDs are not optional if rename and clone semantics are first-class. Path
identity is acceptable only as a legacy bootstrap strategy.

## Simulation 6 — focus / hoist

### Action

The user focuses `Projects / Miku`.

### Expected behavior

- the tree shows only the focused subtree plus its breadcrumb ancestors;
- search and quick-open are scoped to the subtree;
- the user can follow a wikilink to `Concepts / Note Workspace`;
- the outside note opens normally and shows an “outside focus” indicator;
- clearing focus restores the full tree without losing the current note.

### Result

Works if focus is a URL/view state, not a database mutation:

```text
/workspace/note/note-backend?focus=note-miku
```

## Simulation 7 — delete, archive, and stale placements

### Action

The external filesystem removes `references/trilium.md`.

### Expected behavior

- the watcher removes the note projection;
- all placements disappear;
- incoming links become unresolved;
- an open tab shows a recoverable missing-note state;
- the user can remove the stale tab/bookmark;
- no empty reader shell or silent navigation failure occurs.

### Gap discovered

The frontend needs a first-class `note_missing` state. HTTP 404 alone is not
enough because the workspace still contains a selected note and open tabs.

## Simulation 8 — readonly publishing

### Action

The server runs with `MIKU_READONLY=true`.

### Expected behavior

- the editor is unavailable;
- create, rename, move, clone, archive, and delete actions are hidden or
  disabled;
- mutation requests return a clear forbidden response;
- navigation, search, tree, backlinks, and hoisting continue to work;
- the UI does not imply that a disabled action was queued.

## Simulation 9 — rebuild from zero

### Action

Delete every generated index artifact and restart the Rust backend.

### Expected behavior

1. The vault scanner finds every Markdown file.
2. Stable note IDs are loaded from frontmatter.
3. Placements are rebuilt.
4. Links, tags, attributes, mentions, and search documents are rebuilt.
5. The frontend reports indexing progress.
6. The user can navigate notes before the full search projection is complete.

### Result

This is the core ownership test. If the workspace cannot recover from files,
the design has accidentally made the index authoritative.

## Required API invariants exposed by the simulation

1. `note_id` and `placement_id` are distinct.
2. Note mutations and placement mutations are separate operations.
3. File save and index completion are separate observable states.
4. Revision tokens protect writes against external changes.
5. URLs identify note plus optional placement/focus context.
6. Missing notes are a workspace state, not merely a 404 response.
7. Readonly applies to every mutation endpoint, not just the editor button.
8. Rebuild must recover all durable behavior from vault files.

## Implementation blockers found before coding

- Current `PageSummary` cannot represent note identity versus placement.
- Current `IndexReader` cannot query children, placements, attributes, revisions,
  or workspace bootstrap data.
- Current path-based wikilink resolution needs stable-ID/rename behavior.
- Current browser state has no tab, split, placement, focus, or revision model.
- Current SQLite schema should not be expanded incrementally; it should be
  replaced behind a new domain contract.
- The file format needs a documented stable-ID and parent-placement convention.

These are design findings, not implementation failures. They define the first
DDD/TDD boundaries to evaluate after the architecture review.
