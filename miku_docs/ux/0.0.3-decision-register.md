# Miku 0.0.3 decision register

This register prevents implicit architectural choices from disappearing into
implementation. “Decided” means implementation may proceed. “Recommended” is
the current default but should be confirmed before the affected boundary is
built. “Open” requires a product or maintainer choice.

## Decided

| Area | Decision |
| --- | --- |
| Product surface | Separate `miku-web` browser frontend over a Rust backend |
| Frontend | React + TypeScript + Vite |
| Routing | React Router with note, placement, and focus in the URL |
| Server state | TanStack Query |
| Local UI state | Small reducer/store for tabs, splits, panels, focus, and history |
| Editor | CodeMirror 6 Markdown source editor first |
| UI primitives | Radix primitives with Miku-owned CSS tokens and styles |
| Testing | Vitest/Testing Library plus Playwright |
| Backend authority | Markdown files and attachments |
| Domain concepts | Note, Placement, Link, Attribute, Revision |
| Tree identity | `note_id` is distinct from `placement_id` |
| Tree persistence | Parent placement references in note frontmatter |
| Local projection | Memory graph plus Tantivy search; disposable artifacts under `.miku/index/` |
| API contract | Rust publishes OpenAPI; frontend generates TypeScript types |
| Write boundary | Validate, atomic file write, watcher, index, workspace event |
| Readonly | Every mutation endpoint must enforce readonly, not just the UI |
| Production ingress | Caddy is the only public entrypoint; Miku and storage stay internal |

## Recommended defaults

These choices are intentionally conservative and should be used unless a
specific requirement rejects them:

| Area | Default | Why |
| --- | --- | --- |
| Frontend location | Separate sibling project `miku-web` | Keeps the replaceable UI out of the Rust workspace |
| API style | REST/JSON plus SSE index events | Simple to inspect and sufficient for a single-user workspace |
| Note identity | UUIDv4 in frontmatter | Stable across rename and clone; easy to generate in Rust |
| Legacy identity | Relative path until migration/edit | Avoids rewriting an entire existing vault at startup |
| Parent metadata | Ordered `parents` list in child frontmatter | Git-visible and portable without a tree database |
| Search | Tantivy local projection | Rust-native full-text search without SQLite authority |
| Browser support | Current Chrome, Firefox, Safari, and Edge | Matches the desktop/web target without legacy browser cost |
| Styling | CSS modules plus global design tokens | Precise workspace layout without adopting another product's visual system |
| Rich preview | Rust-rendered HTML plus client-side live preview only where needed | Keeps Markdown parsing consistent with backend/indexing |
| Revision model | File hash + mtime initially; content snapshots later | Enough for conflict safety without prematurely designing a history database |
| Frontend package tool | Bun | Matches the repository's existing JavaScript tooling preference |
| Deployment shape | Caddy + Miku Compose profile | TLS and host routing stay at the edge; the app keeps one browser/API origin |

## Open choices

These are the choices I had not exposed clearly and should be decided before
the corresponding implementation begins.

### 1. Frontend repository boundary

Should `miku-web` be:

- a sibling repository next to `miku`;
- a top-level directory in this repository;
- or a Git submodule owned separately?

Recommendation: sibling repository. It makes the frontend genuinely
replaceable and avoids pulling its dependency graph into Rust checks.

### 2. URL identity

Should public URLs use stable IDs or readable paths?

- `/n/note-8f6f0b` — stable through rename;
- `/p/projects/miku/backend` — readable and compatible with v0.0.2;
- both, with canonical ID URLs and path redirects.

Recommendation: support both. Use stable IDs internally and preserve readable
path URLs as redirects/bookmarks.

### 3. Parent metadata format

Should a note's parent references be plain IDs, wikilinks, or a structured
frontmatter object?

Recommendation: structured frontmatter with IDs and optional display aliases;
do not overload ordinary body wikilinks with tree semantics.

### 4. Clone semantics

Should cloning be available in 0.0.3, or only modeled internally first?

Recommendation: model and index multiple placements first; expose clone/move UI
only after rename, delete, and stale-placement behavior is tested.

### 5. Editing model

Should 0.0.3 remain source-editor-first, or require Trilium-like WYSIWYG parity?

Recommendation: source-editor-first. WYSIWYG is a separate product decision
because it can violate Markdown round-tripping.

### 6. Event transport

Should index updates use SSE, WebSocket, or polling?

Recommendation: SSE first. The server publishes one-way invalidation events;
the frontend refetches through TanStack Query. WebSocket is unnecessary until
collaboration or bidirectional live operations exist.

### 7. Multi-user and authentication scope

Should the new API assume one local user, or design for accounts and permissions
now?

Recommendation: single-user trusted boundary for 0.0.3, with an API seam that
does not make adding auth impossible. Do not introduce permission concepts into
the note domain yet.

### 8. Revision durability

Should revisions be Git-only, filesystem snapshots, or a dedicated projection?

Recommendation: Git/file history first; implement content snapshots only after
the core editing workflow is stable.

### 9. Migration policy

Should existing Miku v0.0.2 notes be rewritten with IDs automatically?

Recommendation: no destructive bulk rewrite. Index legacy paths first and offer
an explicit migration command that writes IDs with a reviewable Git diff.

### 10. Backend packaging

Should Rust serve the built frontend in production, or should deployment keep
frontend and backend as separate services?

Recommendation: support both: separate dev servers, one packaged Rust server
that can serve a built `miku-web` artifact. The API boundary remains identical.

## TDD and DDD adoption

Adopt both in a deliberately lightweight form:

- TDD for identity, frontmatter, placement, conflict, atomic-write, and rebuild
  invariants;
- DDD for naming domain concepts and keeping the vault, projection, search, and
  HTTP adapters behind explicit ports;
- no aggregate ceremony, event-sourcing, or repository abstraction until a
  concrete behavior needs it.

The first implementation task should not create React components. It should
write tests for the domain invariants that the workflow simulation exposed.
