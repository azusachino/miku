# ADR-0017 — Web Markdown workspace and file-based note graph

## Decision

Miku 0.0.3 will be designed as a **Trilium-like web workspace over a
Markdown-owned vault**. The frontend becomes a separate React/TypeScript
project. Rust remains responsible for the vault, HTTP API, Markdown parsing,
filesystem watcher, indexing, search, relationship resolution, and safe file
mutations.

The former server-rendered templates and SQLite page-index model are
disposable migration scaffolding. They are not the product contract.

The durable contract is:

1. Markdown files and attachments remain the user's source of truth.
2. Every note is independently inspectable, editable, versionable, and
   removable with ordinary filesystem and Git tools.
3. The index is a rebuildable projection and must not contain irreplaceable
   user knowledge.
4. The UI presents a hierarchical workspace: tree, note workspace, context
   panels, tabs/splits, history, bookmarks, hoisting, and command actions.
5. Note content and tree placement are separate concepts. One Markdown note may
   have multiple placements in the virtual tree without duplicating its body.
6. Frontend code communicates through domain-oriented HTTP APIs, never through
   SQL-shaped payloads or knowledge of the index implementation.

## Domain model

The core domain replaces the page-summary model with these concepts:

```text
Note       canonical Markdown content and frontmatter
Placement  note-to-parent tree edge with ordering and clone semantics
Link       explicit wikilink/embed edge between notes or assets
Attribute  normalized frontmatter/property projection
Revision   file-content hash and revision metadata
```

The canonical file is still the content authority. A stable `id` may be stored
in frontmatter; legacy files without one use their normalized relative path
until a migration or edit assigns an identity.

Example:

```yaml
---
id: note-8f6f0b
title: Rust Backend
type: text
parents:
  - note-miku
  - note-architecture
order: 20
icon: code
---
```

The indexer derives tree edges, backlinks, attributes, tags, mentions, and
search documents from files. It may use memory, Tantivy, or an optional remote
store as an acceleration layer; no particular database is part of the domain
contract. The first local implementation uses an in-memory graph projection
and Tantivy for full-text search. SQLite is not the default product model for
0.0.3.

`id` is generated by Miku when a note is first created or migrated. Duplicate
IDs are an explicit index conflict and are never silently merged. `parents` is
the ordered list of placement parent IDs. Adding a parent creates a placement;
removing a parent removes only that placement. Deleting a note is a separate
operation that removes the canonical Markdown file and all derived placements.
Legacy files without IDs remain path-addressable until migration assigns one.

## Frontend boundary

The new `miku-web` project owns only presentation and ephemeral workspace state:

- tree expansion and selection;
- open tabs and split panes;
- navigation history and scroll positions;
- bookmarks and recent-note presentation;
- command palette state;
- panel widths and responsive drawer state.

The URL is authoritative for the current note and focus scope. Local browser
storage is bounded and disposable. Content, placements, links, attributes, and
revisions are server-owned.

The primary shell is:

```text
launch bar
tree | note tabs/splits | context widgets
status bar
```

The first release reproduces the interaction model of Trilium's tree-centered
workspace without copying its source, assets, or proprietary data format.

### Framework and project boundary

`miku-web` is a separate React/TypeScript project built with Vite. It uses:

- React and React Router for the application and URL-addressable workspace;
- TanStack Query for server state, cache invalidation, and mutations;
- a small reducer/store for tabs, splits, panels, focus, and history;
- CodeMirror 6 for lossless Markdown source editing;
- Radix primitives plus Miku-owned CSS tokens and modules for accessible,
  custom Trilium-like controls;
- Vitest and Testing Library for state/component tests;
- Playwright for browser workflows and visual acceptance;
- Bun for frontend package and script execution.

The frontend is not a Next.js/Remix application. Rust is the only application
backend. The backend publishes an OpenAPI document and the frontend generates
TypeScript API types from that contract.

The frontend state boundary is explicit:

```text
TanStack Query  server state: notes, tree, context, search, revisions
workspace store client state: tabs, splits, panels, focus, selection
URL             current note, placement, focus, heading/scroll location
```

The initial editor is CodeMirror 6 with a React Markdown reader and
optional split view. A block/WYSIWYG editor is deferred until round-trip tests
prove that frontmatter and Markdown semantics are preserved.

### Runtime and ingress boundary

Development may run the Rust service directly on its local port. The deployed
web surface uses Caddy as the single public ingress:

```text
browser -> Caddy (TLS, host routing) -> miku (Rust HTTP server)
                                      -> Markdown vault
                                      -> disposable memory/Tantivy projections
```

Caddy terminates TLS and reverse-proxies to the internal Miku service. The
Miku and database ports are not published publicly by the production Compose
profile. The first deployment keeps the frontend assets served by Miku after
the separate `miku-web` build is produced; a later split static-asset service
must preserve the same browser/API origin and is not required for 0.0.3.

The local development profile remains direct and lightweight. A production
profile may add Caddy without changing the Rust API, vault layout, or frontend
contract.

## Backend boundary

Rust provides domain-oriented endpoints:

```text
GET    /api/v1/workspace
GET    /api/v1/tree?parent_id=...
GET    /api/v1/notes/:id
GET    /api/v1/notes/:id/context
GET    /api/v1/notes/:id/children
GET    /api/v1/notes/:id/backlinks
GET    /api/v1/search?q=...
GET    /api/v1/events

POST   /api/v1/notes
PATCH  /api/v1/notes/:id
POST   /api/v1/notes/:id/placements
DELETE /api/v1/placements/:id
POST   /api/v1/notes/:id/bookmark
POST   /api/v1/notes/:id/archive
```

All writes go through the filesystem service. The write path is:

```text
validate → write temporary file → flush/fsync → atomic rename
         → watcher event → parse/index → workspace event
```

Readonly mode disables or rejects mutation endpoints. An index lag is visible
as status, never silently treated as a successful durable write.

## Indexing strategy

The indexer is a projection pipeline:

```text
filesystem event
  → debounce/coalesce
  → parse Markdown/frontmatter
  → resolve note identity
  → rebuild placements, links, attributes, tags, mentions
  → update search projection
  → publish index/workspace event
```

The rebuild command must be able to delete every derived index artifact and
reconstruct the workspace from the vault. Search and graph performance are
implementation concerns, not reasons to move authority into a database.

The local projection layout is disposable:

```text
.miku/index/     Tantivy segments and rebuild metadata
memory           live tree, links, attributes, and workspace graph
vault files      notes, frontmatter, attachments, and revisions
```

The `.miku/index/` directory may be deleted at any time. It is never edited by
users and is never required to recover note content or placements.

## 0.0.3 delivery order

1. Scaffold `miku-web` and build the shell against fixture data.
2. Test tree selection, tabs, splits, history, command actions, and hoisting.
3. Define Rust domain types, file/frontmatter representation, and OpenAPI.
4. Expose read-only workspace/tree/note/context/search APIs.
5. Connect the frontend through generated API types and TanStack Query.
6. Replace the SQLite-specific page index with the new graph/search projection.
7. Add CodeMirror editing and revision-safe file writes.
8. Add create/rename/move/clone/archive/revision workflows.
9. Add accessibility, responsive behavior, performance, and browser acceptance
   coverage.

Graph maps, scripting, AI panels, sync, encryption, collaboration, native
mobile, collections, and WYSIWYG parity are not prerequisites for this
milestone.

The implementation is tracked in the frontend source and browser acceptance
checks.

## Verification

The design must pass the live browser acceptance workflow before implementation
is considered ready. The most important end-to-end proof is:

```text
open vault → select tree note → open clone placement → edit Markdown
→ watcher/index update → inspect backlink/context → restart
→ rebuild projection → recover the same workspace from files
```

## Why

The former frontend demonstrated that server-rendered HTML can
display Markdown, but it does not provide the continuous tree-centered
workspace that motivated the Trilium comparison. Its current `PageSummary` and
SQLite tables also encode a flat page index rather than a note/placement model.

Tolaria demonstrates a modern files-first vault with keyboard navigation and
frontmatter conventions. SilverBullet demonstrates a browser-native Markdown
workspace and page picker. Trilium provides the target interaction language:
hierarchical navigation, focused note work, hoisting, bookmarks, context
actions, and workspace continuity.

The synthesis keeps the ownership boundary from Miku, the browser surface from
SilverBullet, the files-first conventions from Tolaria, and the workspace
interaction model from Trilium.

## Trade-offs / Rejected

- **Keep the current server-rendered frontend** — rejected. It optimizes for a
  reader page, while 0.0.3 needs a stateful workspace with tabs, splits, tree
  selection, and coordinated panels.
- **Keep SQLite as the domain model** — rejected. A relational projection can
  remain an implementation option, but note identity and placement semantics
  must be expressed in the domain and reconstructible from files.
- **Persist a separate tree manifest first** — rejected for 0.0.3. Parent
  placement metadata lives with the child note in frontmatter so moving or
  cloning a note remains visible to Git and ordinary editors. A manifest may be
  reconsidered only if frontmatter becomes unmanageable at scale.
- **Use physical folders as the complete tree model** — rejected. Physical
  paths cannot represent multiple placements or clones. They remain a useful
  default organization and import/export representation.
- **Copy Trilium's source or assets** — rejected. Study behavior and structure;
  independently implement Miku's UI under the applicable licenses.
- **Start with plugins, AI, or maps** — rejected. They would obscure whether
  the core workspace navigation and file/index boundary work.
